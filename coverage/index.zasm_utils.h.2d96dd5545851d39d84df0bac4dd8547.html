60 // Returns a control flow graph - splits the given range into basic blocks (which only possibly transfers
61 // to another block as their last instruction), and marks all possible blocks each block may transfer
62 // control to.
63 //
64 // For example:
65 //
66 // An "if" control flow will produce a basic block that goes to one of two other blocks - either following
67 // its "if" condition being true and transfering controll to the corresponding block, or by failing that
68 // condition and transfering control to the next block by "falling through" (doing nothing, really).
69 //
70 // The resulting control flow graph should be reducible, which basically means it does not contain
71 // any loops with more than one entrypoint. That only happens with:
72 // 1) unstructured control flow constructs like goto
73 // 2) intense compiler optimizations manipulating the control flow
74 // Neither should be the case for the ZASM we observe from the zscript compiler, which greatly simplifies things
75 // as we don't need to convert the graph to be reducible.
76 //
77 // https://www2.cs.arizona.edu/~collberg/Teaching/453/2009/Handouts/Handout-15.pdf
78 ZasmCFG zasm_construct_cfg(const zasm_script* script, std::vector<std::pair<pc_t, pc_t>> pc_ranges);
