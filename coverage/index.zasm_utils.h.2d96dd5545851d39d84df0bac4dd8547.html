103 // Returns a control flow graph - splits the given range into basic blocks (which only possibly transfers
104 // to another block as their last instruction), and marks all possible blocks each block may transfer
105 // control to.
106 //
107 // For example:
108 //
109 // An "if" control flow will produce a basic block that goes to one of two other blocks - either following
110 // its "if" condition being true and transfering controll to the corresponding block, or by failing that
111 // condition and transfering control to the next block by "falling through" (doing nothing, really).
112 //
113 // The resulting control flow graph should be reducible, which basically means it does not contain
114 // any loops with more than one entrypoint. That only happens with:
115 // 1) unstructured control flow constructs like goto
116 // 2) intense compiler optimizations manipulating the control flow
117 // Neither should be the case for the ZASM we observe from the zscript compiler, which greatly simplifies things
118 // as we don't need to convert the graph to be reducible.
119 //
120 // https://www2.cs.arizona.edu/~collberg/Teaching/453/2009/Handouts/Handout-15.pdf
121 ZasmCFG zasm_construct_cfg(const zasm_script* script, std::vector<std::pair<pc_t, pc_t>> pc_ranges);
