74 // Returns a control flow graph - splits the given range into basic blocks (which only possibly transfers
75 // to another block as their last instruction), and marks all possible blocks each block may transfer
76 // control to.
77 //
78 // For example:
79 //
80 // An "if" control flow will produce a basic block that goes to one of two other blocks - either following
81 // its "if" condition being true and transfering controll to the corresponding block, or by failing that
82 // condition and transfering control to the next block by "falling through" (doing nothing, really).
83 //
84 // The resulting control flow graph should be reducible, which basically means it does not contain
85 // any loops with more than one entrypoint. That only happens with:
86 // 1) unstructured control flow constructs like goto
87 // 2) intense compiler optimizations manipulating the control flow
88 // Neither should be the case for the ZASM we observe from the zscript compiler, which greatly simplifies things
89 // as we don't need to convert the graph to be reducible.
90 //
91 // https://www2.cs.arizona.edu/~collberg/Teaching/453/2009/Handouts/Handout-15.pdf
92 ZasmCFG zasm_construct_cfg(const zasm_script* script, std::vector<std::pair<pc_t, pc_t>> pc_ranges);
